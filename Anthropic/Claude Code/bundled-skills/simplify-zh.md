`/simplify → 4 个清理代理并行 → 应用修复`

你正在提高更改代码的质量，而不是寻找错误。审查
它是否存在重用、简化、效率和高度问题，然后修复你
发现的内容。不要寻找正确性错误——那是 `/code-review` 的用途。

## 阶段 0 — 收集差异

运行 `git diff @{upstream}...HEAD`（或 `git diff main...HEAD` / `git diff HEAD~1`
如果没有上游）以获取审查下的统一差异。如果有
未提交的更改，或范围差异为空，还要运行 `git diff HEAD` 并
将工作树更改包括在范围内——审查通常在提交之前
运行。如果传递了 PR 编号、分支名称或文件路径作为参数，
则审查该目标。将此差异视为审查范围。

## 阶段 1 — 审查（4 个清理代理并行）

通过 Agent 工具启动 **4 个独立的审查代理**，全部在
单个消息中，以便它们并发运行。将差异和以下四个角度之一传递给每个代理。每个代理返回其发现，带有 `file`、`line`、一个
一行 `summary` 和具体成本（重复、浪费或
更难维护的内容）。

### 重用

标记重新实现代码库已有内容的新代码——Grep 共享/实用模块和更改附近的文件，
并命名要调用的现有帮助器。

### 简化

标记差异添加的不必要复杂性：冗余或可推导状态、
略有变化的复制粘贴、深度嵌套、遗留的死代码。命名
执行相同工作的更简单形式。

### 效率

标记差异引入的浪费工作：冗余计算或重复 I/O、
顺序运行的独立操作、添加到启动或
热路径的阻塞工作。命名更便宜的替代方案。

### 高度

检查每个更改是否在正确的深度实现，而不是作为脆弱的
创可贴。共享基础设施上的分层特殊情况是修复
不够深入的迹象——更喜欢概括底层机制而不是添加
特殊情况。

## 阶段 2 — 应用修复

等待所有四个代理完成，去重指向同一行或机制的发现，
并直接修复每个剩余的发现。跳过任何修复会改变预期行为、需要在审查差异之外进行大量更改，或者你判断为误报的发现——注意跳过而不是与其争论。以简要总结修复的内容和跳过的内容结束（或确认代码已经干净）。
