name: init
description: 使用代码库文档初始化新的 CLAUDE.md 文件。
---

为此仓库设置最小的 CLAUDE.md（以及可选的技能和钩子）。CLAUDE.md 被加载到每个 Claude Code 会话中，因此它必须简洁——仅包括 Claude 没有它会出错的内容。

## 阶段 0：检查现有的 CLAUDE.md

在询问任何内容之前，检查项目根目录是否已存在 CLAUDE.md（只需 `cat ./CLAUDE.md`——只有项目根文件才算；暂不探索树）。这将分支阶段 1。

## 阶段 1：询问要设置什么

使用 AskUserQuestion 找出用户想要什么。你问哪个问题取决于阶段 0。仅使用 **Q1** 调用 AskUserQuestion——不要在同一调用中包括 Q2。只有在看到 Q1 答案后才问 Q2，因为"让 Claude 决定"会跳过它。

在第一个问题之前，将此入门作为正常助手文本打印，以便首次用户了解术语：

> 快速上下文：
> - **CLAUDE.md** 文件为 Claude 提供项目、个人工作流程或组织的持久指令。Claude 在每个会话开始时读取它们。
> - **技能**是 Claude 在任务匹配时自动调用的打包指令，或者你用斜杠命令触发（例如 `/frontend-design`、`/commit-push-pr`）。
> - **钩子**允许你在生命周期事件上自动运行 shell 命令：在 Claude 被你的输入阻止时获得通知，编辑后自动格式化，提交前强制检查——这些是确定性的，Claude 无法跳过它们。

**如果 CLAUDE.md 已存在**，问：
- "我找到了现有的 CLAUDE.md。你想做什么？"
  选项："审查并改进它" | "保留它，设置其他东西" | "重新开始（替换它）"
  改进的描述："探索代码库中的变化并建议对现有文件进行有针对性的编辑。"
  保留的描述："跳过 CLAUDE.md。直接进入技能和钩子。"
  重新开始的描述："丢弃它并写入新文件。"
  路由：
  - "审查并改进" → 跳过 Q1/Q2；探索（阶段 2），问单个阶段 3-lite 问题，然后进入阶段 4 的差异提案，然后阶段 8。
  - "保留它" → 跳过 Q1，问 Q2（将其第四个选项重命名为"两者都不——跳过设置"）。如果他们选择"两者都不——跳过设置"，直接跳到阶段 8："无需设置——你的 CLAUDE.md 未更改。"否则：阶段 2 → 阶段 3 提案（无差距填充访谈）→ 按队列的阶段 6/7 → 阶段 8。对于阶段 7 的钩子目标文件默认值，将此路径视为"项目"（`.claude/settings.json`）。
  - "重新开始" → 继续到下面的 Q1，就像没有文件存在一样。

**如果不存在 CLAUDE.md**（或用户选择了"重新开始"），问：
- Q1："应该 /init 设置哪些 CLAUDE.md 文件？"
  选项："项目 CLAUDE.md" | "个人 CLAUDE.local.md" | "项目 + 个人两者" | "让 Claude 决定"
  项目的描述："签入源控制的团队共享指令——架构、编码标准、常见工作流程。"
  个人的描述："你对此项目的私人偏好（gitignored，不共享）——你的角色、沙箱 URL、首选测试数据、工作流程怪癖。"
  让 Claude 决定的描述："最快路径——项目 CLAUDE.md 加上适合此仓库的任何技能或钩子。没有后续问题；你会在写入之前批准所有内容。"
  如果用户选择"让 Claude 决定"，跳过 Q2——将其视为没有技能/钩子约束的项目 CLAUDE.md。

- Q2："还要设置技能和钩子吗？"
  选项："技能 + 钩子" | "仅技能" | "仅钩子" | "两者都不，仅 CLAUDE.md"
  技能的描述："Claude 在任务匹配时自动调用的打包指令，或者你用斜杠命令触发（例如 `/frontend-design`、`/commit-push-pr`）。"
  钩子的描述："在工具事件上运行的确定性 shell 命令（例如，每次编辑后格式化）。Claude 无法跳过它们。"
  Q2 是提示，不是过滤器——阶段 3 提出适合代码库的内容并注意任何偏差。

## 阶段 2：探索代码库

启动子代理调查代码库，并要求它读取关键文件以了解项目：清单文件（package.json、Cargo.toml、pyproject.toml、go.mod、pom.xml 等）、README、Makefile/构建配置、CI 配置、现有 CLAUDE.md、.claude/rules/、AGENTS.md、.cursor/rules 或 .cursorrules、.github/copilot-instructions.md、.windsurfrules、.clinerules、.mcp.json。

检测：
- 构建、测试和 lint 命令（特别是非标准的）
- 语言、框架和包管理器
- 项目结构（带有工作区的 monorepo、多模块或单项目）
- 与语言默认值不同的代码风格规则
- 非明显的陷阱、必需的环境变量或工作流程怪癖
- 现有的 .claude/skills/ 和 .claude/rules/ 目录
- 格式化程序配置（prettier、biome、ruff、black、gofmt、rustfmt 或统一格式脚本，如 `npm run format` / `make fmt`）
- Git worktree 使用：运行 `git worktree list` 检查此仓库是否有多个 worktree（仅当用户想要个人 CLAUDE.local.md 时相关）

注意你无法仅从代码中弄清楚的内容——这些成为访谈问题。

## 阶段 3：填补差距

使用 AskUserQuestion 收集你仍然需要编写好的 CLAUDE.md 文件和技能的内容。仅询问代码无法回答的内容。

如果用户选择了项目 CLAUDE.md、两者或"让 Claude 决定"：询问代码库实践——非明显的命令、陷阱、分支/PR 约定、必需的环境设置、测试怪癖。跳过已在 README 中或从清单文件中明显的内容。不要将任何选项标记为"推荐"——这是关于他们的团队如何工作，而不是最佳实践。

如果用户选择了个人 CLAUDE.local.md 或两者：询问他们，而不是代码库。不要将任何选项标记为"推荐"——这是关于他们的个人偏好，而不是最佳实践。问题示例：
  - 他们在团队中的角色是什么？（例如，"后端工程师"、"数据科学家"、"新员工入职"）
  - 他们对此代码库及其语言/框架有多熟悉？（以便 Claude 可以校准解释深度）
  - 他们是否有个人沙箱 URL、测试帐户、API 密钥路径或 Claude 应该知道的本地设置详细信息？
  - 仅当阶段 2 发现多个 git worktree 时：询问他们的 worktree 是否嵌套在主仓库内（例如，`.claude/worktrees/<name>/`）或同级/外部（例如，`../myrepo-feature/`）。如果嵌套，向上文件遍历会自动找到主仓库的 CLAUDE.local.md——无需特殊处理。如果是同级/外部，个人内容应该位于主目录文件中（例如，`~/.claude/<project-name>-instructions.md`），每个 worktree 获得一个导入它的单行 CLAUDE.local.md 存根：`@~/.claude/<project-name>-instructions.md`。永远不要将此导入放在项目 CLAUDE.md 中——那会将个人引用检查到团队共享文件中。
  - 任何沟通偏好？（例如，"简洁"、"总是解释权衡"、"最后不要总结"）

如果用户在阶段 0 选择了"审查并改进"：只问一个问题——"自编写此 CLAUDE.md 以来，团队的工作方式是否发生了任何变化（新约定、命令、陷阱）？"，选项为"不，没有变化" | "是——让我描述"。如果他们选择是，在继续之前询问发生了什么变化（自由文本）。然后跳到阶段 4。

**从阶段 2 发现和差距填充答案综合提案。** 对于每个项目，选择适合证据的工件类型：

  - **钩子**——确定性、快速、每次编辑的 shell 命令（格式化、linting 更改的文件）。
  - **技能**——按需多步骤工作流程（`/verify`、`/deploy-staging`、会话报告）。
  - **CLAUDE.md 注释**——塑造行为但不强制执行的指导（约定、沟通风格）。

包括 Q1 暗示的 CLAUDE.md 文件（项目、个人、两者或"让 Claude 决定"→项目）作为提案的第一个要点，以及每个将涵盖的内容的单行摘要。然后列出技能/钩子/注释。在"保留它"路径上，省略 CLAUDE.md 文件要点和注释（阶段 4 不会运行）。在 Q1 = 仅个人的"重新开始"路径上，添加一个要点，注意现有的项目 CLAUDE.md 将保持不变（他们选择不将其替换为项目文件）。

提出适合的内容。如果用户给出了 Q2 提示并且你的提案偏离了它（例如，他们说"仅钩子"但不存在钩子形状的内容），在提案顶部用一行说明，并仍然提出更适合的工件。

**将提案作为正常助手文本打印**，每个项目一个要点：

> 这是我要设置的内容：
> • **[工件类型：文件/钩子/技能/注释]** — [单行描述]
> • …

然后使用简单问题调用 AskUserQuestion（"这看起来对吗？"），选项如"看起来不错——继续" | "删除钩子" | "删除技能"。不要使用 `preview` 字段——提案在滚动中已经可见。该工具自动添加"其他"选项以进行自定义调整。

**从接受的提案构建偏好队列。** 每个条目：{类型：钩子|技能|注释、描述、目标文件、任何阶段 2 来源的详细信息，如实际测试/格式命令}。阶段 6 和阶段 7 的钩子子要点消耗此队列；阶段 4/5 直接根据批准提案的文件要点进行门控；阶段 7 的 GitHub-CLI 和 linting 检查无论队列内容如何都运行。

## 阶段 4：写入 CLAUDE.md（如果批准的提案包括它，或在"审查并改进"路径上）

在项目根目录写入最小的 CLAUDE.md。每一行都必须通过此测试："删除此会导致 Claude 犯错吗？"如果否，则删除。

如果用户在阶段 0 选择了"审查并改进它"：不要重新写入——读取现有文件，与阶段 2 发现和阶段 3-lite 答案进行比较，并建议具体的添加/删除作为差异，每个差异都有一个单行原因。现有文件是基线；你的工作是捕获缺失、过时或臃肿的内容。打印差异后，在写入任何内容之前调用 AskUserQuestion（"应用这些编辑？"，选项如"应用全部" | "让我选择哪些" | "跳过——保持原样"）。

**消耗阶段 3 偏好队列中目标是 CLAUDE.md 的 `note` 条目**（团队级注释）——将每个作为最相关部分中的简洁行添加。这些是用户希望 Claude 遵循但不需要保证的行为（例如，"在实现之前提出计划"、"重构时解释权衡"）。将针对个人的注释留给阶段 5。

包括：
- Claude 无法猜测的构建/测试/lint 命令（非标准脚本、标志或序列）
- 与语言默认值不同的代码风格规则（例如，"优先使用 type 而不是 interface"）
- 测试指令和怪癖（例如，"使用以下命令运行单个测试：pytest -k 'test_name'"）
- 仓库礼仪（分支命名、PR 约定、提交风格）
- 必需的环境变量或设置步骤
- 非明显的陷阱或架构决策
- 现有 AI 编码工具配置的重要部分（如果存在）（AGENTS.md、.cursor/rules、.cursorrules、.github/copilot-instructions.md、.windsurfrules、.clinerules）

排除：
- 逐文件结构或组件列表（Claude 可以通过读取代码库发现这些）
- Claude 已经知道的标准语言约定
- 通用建议（"编写干净的代码"、"处理错误"）
- 详细的 API 文档或长参考——改用 `@path/to/import` 语法（例如，`@docs/api-reference.md`）按需内联内容而不臃肿 CLAUDE.md
- 频繁变化的信息——使用 `@path/to/import` 引用源，以便 Claude 始终读取当前版本
- 长教程或演练（移动到单独文件并使用 `@path/to/import` 引用，或放在技能中）
- 从清单文件中明显的命令（例如，标准"npm test"、"cargo test"、"pytest"）

具体："在 TypeScript 中使用 2 空格缩进"比"正确格式化代码"更好。

不要重复自己，不要编造"常见开发任务"或"开发提示"等部分——仅包括在你读取的文件中明确找到的信息。

在文件前加上：

```
# CLAUDE.md

此文件为 Claude Code (claude.ai/code) 在此仓库中处理代码时提供指导。
```

对于具有多个关注点的项目，建议将指令组织到 `.claude/rules/` 中作为单独的专注文件（例如，`code-style.md`、`testing.md`、`security.md`）。这些与 CLAUDE.md 一起自动加载，并且可以使用 `paths` 前言限定为特定文件路径。

对于具有不同子目录的项目（monorepo、多模块项目等）：提及可以添加子目录 CLAUDE.md 文件以用于模块特定指令（当 Claude 在这些目录中工作时自动加载）。如果用户想要，提议创建它们。

## 阶段 5：写入 CLAUDE.local.md（如果批准的提案包括它）

在项目根目录写入最小的 CLAUDE.local.md。此文件与 CLAUDE.md 一起自动加载。创建后，将 `CLAUDE.local.md` 添加到项目的 .gitignore，以便保持私有。

**消耗阶段 3 偏好队列中目标是 CLAUDE.local.md 的 `note` 条目**（个人级注释）——将每个作为简洁行添加。如果用户在阶段 1 选择了仅个人，这是注释条目的唯一消费者。

包括：
- 用户在团队中的角色以及对代码库的熟悉程度（以便 Claude 可以校准解释）
- 个人沙箱 URL、测试帐户或本地设置详细信息
- 个人工作流程或沟通偏好

保持简短——仅包括会使 Claude 对此用户的响应明显更好的内容。

如果阶段 2 发现多个 git worktree 并且用户确认他们使用同级/外部 worktree（不是嵌套在主仓库内）：向上文件遍历不会从所有 worktree 找到单个 CLAUDE.local.md。将实际个人内容写入 `~/.claude/<project-name>-instructions.md` 并使 CLAUDE.local.md 成为导入它的单行存根：`@~/.claude/<project-name>-instructions.md`。用户可以将此单行存根复制到每个同级 worktree。永远不要将此导入放在项目 CLAUDE.md 中。如果 worktree 嵌套在主仓库内（例如，`.claude/worktrees/`），则无需特殊处理——主仓库的 CLAUDE.local.md 会自动找到。

如果 CLAUDE.local.md 已存在：读取它，建议具体的添加，不要静默覆盖。

## 阶段 6：建议并创建技能（如果批准的提案包括任何）

技能添加 Claude 可以按需使用的能力，而不会使每个会话臃肿。

**首先，消耗阶段 3 偏好队列中的 `skill` 条目。** 每个排队的技能偏好成为针对用户描述的 SKILL.md。对于每个：
- 从偏好命名它（例如，"verify-deep"、"session-report"、"deploy-sandbox"）
- 使用用户在访谈中的自己的词加上阶段 2 发现的任何内容（测试命令、报告格式、部署目标）编写正文。如果偏好映射到现有的捆绑技能（例如，`/verify`），编写一个项目技能，在其之上添加用户的特定约束——告诉用户捆绑技能仍然存在，他们的是附加的。
- 如果偏好未充分指定，请快速跟进（例如，"verify-deep 应该运行哪个测试命令？"）

**然后建议额外技能**，超出队列，当你发现：
- 特定任务的参考知识（子系统的约定、模式、风格指南）
- 用户希望直接触发的可重复工作流程（部署、修复问题、发布过程、验证更改）

对于每个建议的技能，提供：名称、单行目的以及为什么它适合此仓库。

如果 `.claude/skills/` 已存在技能，请先审查它们。不要覆盖现有技能——仅提出补充现有内容的新技能。

在 `.claude/skills/<skill-name>/SKILL.md` 创建每个技能：

```yaml
---
name: <skill-name>
description: <技能做什么以及何时使用它>
---

<给 Claude 的指令>
```

默认情况下，用户（`/<skill-name>`）和 Claude 都可以调用技能。对于具有副作用的技能（例如，`/deploy`、`/fix-issue 123`），添加 `disable-model-invocation: true`，以便只有用户可以触发它，并使用 `$ARGUMENTS` 接受输入。

## 阶段 7：建议额外优化

告诉用户，既然 CLAUDE.md 和技能（如果选择）已就位，你现在将建议一些额外的优化。

检查环境并询问你发现的每个差距（使用 AskUserQuestion）：

- **GitHub CLI**：运行 `which gh`（或在 Windows 上运行 `where gh`）。如果它缺失并且项目使用 GitHub（检查 `git remote -v` 是否为 github.com），询问用户是否要安装它。解释 GitHub CLI 让 Claude 可以直接帮助提交、拉取请求、问题和代码审查。

- **Linting**：如果阶段 2 未发现 lint 配置（对于项目的语言没有 .eslintrc、ruff.toml、.golangci.yml 等），询问用户是否希望 Claude 为此代码库设置 linting。解释 linting 可以早期发现问题，并为 Claude 自己的编辑提供快速反馈。

- **提案来源的钩子**（如果批准的提案包括任何）：消耗阶段 3 偏好队列中的 `hook` 条目。如果阶段 2 发现了格式化程序并且队列没有格式化钩子，则提供编辑时格式化作为后备。

  对于每个钩子偏好（来自队列或格式化程序后备）：

  1. 目标文件：基于阶段 1 CLAUDE.md 选择的默认值——项目 → `.claude/settings.json`（团队共享，已提交）；个人 → `.claude/settings.local.json`。仅当用户在阶段 1 选择了"两者"或偏好不明确时才询问。为所有钩子询问一次，而不是每个钩子。

  2. 从偏好中选择事件和匹配器：
     - "每次编辑后" → `PostToolUse`，匹配器 `Write|Edit`
     - "当 Claude 完成时" / "在我审查之前" → `Stop` 事件（在每个回合结束时触发——包括只读的）
     - "在运行 bash 之前" → `PreToolUse`，匹配器 `Bash`
     - "在提交之前"（字面 git-commit 门控）→ **不是 hooks.json 钩子。** 匹配器无法按命令内容过滤 Bash，因此无法仅针对 `git commit`。将其路由到 git pre-commit 钩子（`.git/hooks/pre-commit`、husky、pre-commit 框架）——提议编写一个。如果用户实际上是指"在我审查和提交 Claude 的输出之前"，那是 `Stop`——探针以消除歧义。
     如果偏好不明确，则探针。

  3. **加载钩子参考**（每次 `/init` 运行一次，在第一个钩子之前）：使用 `skill: 'update-config'` 和以 `[hooks-only]` 开头的参数调用 Skill 工具，后跟你正在构建的内容的单行摘要——例如，`[hooks-only] 使用 ruff 为 .claude/settings.json 构造 PostToolUse/Write|Edit 格式钩子`。这将钩子架构和验证流程加载到上下文中。后续钩子重用它——不要重新调用。

  4. 遵循技能的**"构造钩子"**流程：去重检查 → 为此项目构造 → 管道测试原始 → 包装 → 写入 JSON → `jq -e` 验证 → 实时证明（对于可触发匹配器上的 `Pre|PostToolUse`）→ 清理 → 交接。目标文件和事件/匹配器来自上面的步骤 1-2。

  在继续之前对每个"是"采取行动。
