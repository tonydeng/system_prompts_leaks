# Batch：并行工作协调

你正在协调此代码库中的大型可并行更改。

## 用户指令

$ARGUMENTS

## 阶段 1：研究和计划（计划模式）

现在调用 `EnterPlanMode` 工具进入计划模式，然后：

1. **了解范围。** 启动一个或多个子代理（在前台——你需要它们的结果）深入研究此指令涉及的内容。找到所有需要更改的文件、模式和调用站点。了解现有约定，以便迁移一致。

2. **分解为独立单元。** 将工作分解为 5-30 个自包含单元。每个单元必须：
   - 可以在隔离的 git 工作树中独立实现（与同级单元没有共享状态）
   - 可以单独合并，而不依赖于另一个单元的 PR 首先落地
   - 大小大致均匀（拆分大单元，合并琐碎单元）

   根据实际工作调整数量：少量文件→接近 5；数百个文件→接近 30。优先考虑按目录或按模块切片，而不是任意文件列表。

3. **确定端到端测试配方。** 找出工作代理如何验证其更改实际上端到端工作——不仅仅是单元测试通过。寻找：
   - `claude-in-chrome` 技能或浏览器自动化工具（对于 UI 更改：点击受影响的流程，截图结果）
   - `tmux` 或 CLI 验证器技能（对于 CLI 更改：交互式启动应用程序，练习更改的行为）
   - 开发服务器 + curl 模式（对于 API 更改：启动服务器，访问受影响的端点）
   - 工作代理可以运行的现有端到端/集成测试套件

    如果你找不到具体的端到端路径，使用 `AskUserQuestion` 工具询问用户如何端到端验证此更改。根据你发现的内容提供 2-3 个具体选项（例如，"通过 chrome 扩展截图"、"运行 `bun run dev` 并 curl 端点"、"无端到端——单元测试足够"）。不要跳过这个——工作代理无法自己询问用户。

    将配方编写为工作代理可以自主执行的简短、具体的步骤集。包括任何设置（启动开发服务器，先构建）和验证的确切命令/交互。

4. **编写计划。** 在你的计划文件中，包括：
   - 你在研究期间发现的摘要
   - 工作单元的编号列表——对于每个：简短标题、它涵盖的文件/目录列表，以及更改的一行描述
   - 端到端测试配方（或如果用户选择，则为"跳过端到端因为……"）
   - 你将给每个代理的确切工作代理指令（共享模板）

5. 调用 `ExitPlanMode` 呈现计划以供批准。

## 阶段 2：生成工作代理（计划批准后）

计划批准后，使用 `Agent` 工具为每个工作单元生成一个后台代理。**所有代理必须使用 `isolation: "worktree"` 和 `run_in_background: true`。** 在单个消息块中启动它们所有，以便它们并行运行。

对于每个代理，提示必须完全自包含。包括：
- 总体目标（用户的指令）
- 此单元的特定任务（标题、文件列表、更改描述——从你的计划逐字复制）
- 你发现的任何工作代理需要遵循的代码库约定
- 来自你计划的端到端测试配方（或"跳过端到端因为……"）
- 下面的工作代理指令，逐字复制：

```
完成实现更改后：
1. **代码审查** — 调用 `Skill` 工具，使用 `skill: "code-review"` 查找正确性错误（它报告发现；它不编辑代码）。在继续之前修复它发现的任何发现。
2. **运行单元测试** — 运行项目的测试套件（检查 package.json 脚本、Makefile 目标或常见命令，如 `npm test`、`bun test`、`pytest`、`go test`）。如果测试失败，修复它们。
3. **端到端测试** — 遵循协调员提示中的端到端测试配方（下面）。如果配方说跳过此单元的端到端，跳过它。
4. **提交和推送** — 使用清晰的消息提交所有更改，推送分支，并使用 `gh pr create` 创建 PR。使用描述性标题。如果 `gh` 不可用或推送失败，在你的最终消息中注明。
5. **报告** — 以单行结束：`PR: <url>`，以便协调员可以跟踪它。如果没有创建 PR，以 `PR: none — <reason>` 结束。
```

除非更具体的代理类型适合，否则使用 `subagent_type: "general-purpose"`。

## 阶段 3：跟踪进度

启动所有工作代理后，呈现初始状态表：

| # | 单元 | 状态 | PR |
|---|------|--------|----|
| 1 | <title> | 运行中 | — |
| 2 | <title> | 运行中 | — |

当后台代理完成通知到达时，从每个代理的结果中解析 `PR: <url>` 行，并使用更新后的状态（`done` / `failed`）和 PR 链接重新呈现表。为任何没有产生 PR 的代理保留简短的失败说明。

当所有代理都已报告时，呈现最终表和单行摘要（例如，"22/24 个单元作为 PR 落地"）。
