`medium effort → 3+5 angles × 6 candidates → 1-vote verify → ≤8 findings`

你正在进行**中等精确度**审查：你提出的每个发现都应该是维护者会采取行动的。

## 阶段 0 — 收集 diff

运行 `git diff @{upstream}...HEAD`（如果没有上游，则运行 `git diff main...HEAD` / `git diff HEAD~1`）以获取正在审查的统一 diff。如果有未提交的更改，或者范围 diff 为空，还要运行 `git diff HEAD` 并将工作树更改包含在审查范围内——审查通常在提交之前运行。如果传递了 PR 编号、分支名称或文件路径作为参数，则改为审查该目标。将此 diff 视为审查范围。

## 阶段 1 — 查找候选（3 个正确性角度 + 3 个清理角度 + 1 个高度角度 + 1 个约定角度，每个最多 6 个）

通过 Agent 工具运行 **8 个独立的查找角度**。每个角度找出**最多 6 个候选发现**，包含 `file`（文件）、`line`（行号）、单行 `summary`（摘要）和具体的 `failure_scenario`（失败场景）。

### 角度 A — 逐行 diff 扫描

逐行读取 diff 中的每个 hunk。然后为每个 hunk 读取包含它的函数——被触及函数中未更改行中的 bug 也在范围内（PR 重新暴露了它们或未能修复它们）。对于每一行，问：什么输入、状态、时机或平台会使这一行出错？查找倒置/错误的条件、差一错误、null/undefined 解引用、缺失的 `await`、假值零检查、错误的变量复制粘贴、catch 中吞掉的错误、未转义的 regex 元字符。

### 角度 B — 移除行为审计器

对于 diff 删除或替换的每一行，命名它强制执行的不变量或行为，然后在新代码中搜索该不变量在哪里重新建立。如果找不到，那就是一个候选：一个被移除的守卫、一个被丢弃的错误路径、一个缩小的验证、一个被删除的覆盖真实情况的测试。

### 角度 C — 跨文件追踪器

对于 diff 更改的每个函数，找到它的调用者（用 Grep 搜索符号）并检查更改是否破坏了任何调用点：新的前置条件、更改的返回形状、新的异常、时序/排序依赖。还要检查被调用者：同一 PR 中的并行更改是否使调用变得不安全？

### 复用

上述角度寻找 bug；这个角度和接下来的两个角度在更改的代码中寻找清理。标记重新实现代码库已有内容的新代码——用 Grep 搜索共享/实用工具模块和更改附近的文件，并命名要调用的现有辅助函数。

### 简化

标记 diff 添加的不必要的复杂性：冗余或可推导的状态、略有变化的复制粘贴、深层嵌套、遗留的死代码。命名执行相同工作的更简单形式。

### 效率

标记 diff 引入的浪费工作：冗余计算或重复 I/O、顺序运行的独立操作、添加到启动或热路径的阻塞工作。还要标记从闭包或捕获环境构建的长生命周期对象——它们在整个对象生命周期内保持整个包含范围存活（当该范围持有大值时会导致内存泄漏）；更喜欢只复制所需字段的类/结构体。命名更便宜的替代方案。

### 高度

检查每个更改是否在正确的深度实现，而不是作为脆弱的补丁。在共享基础设施上分层添加的特殊情况表明修复不够深入——更喜欢泛化底层机制而不是添加特殊情况。

### 约定（CLAUDE.md）

找到管理更改代码的 CLAUDE.md 文件：用户级别的 `~/.claude/CLAUDE.md`、仓库根目录的 CLAUDE.md，加上作为更改文件祖先的目录中的任何 CLAUDE.md 或 CLAUDE.local.md（目录的 CLAUDE.md 仅适用于该目录或其下的文件）。读取每个存在的文件，然后检查 diff 是否存在对其所述规则的明显违反。

只有当你能引用确切的规则和违反它的确切行时，才标记违规——没有风格偏好，没有模糊的"文档精神"推断。在发现中，命名 CLAUDE.md 路径并引用规则，以便报告可以引用它。如果没有适用的 CLAUDE.md，则此角度不返回任何内容。

清理、高度和约定候选使用相同的 `file`/`line`/`summary` 形式；在 `failure_scenario` 中，陈述具体成本（什么是重复的、浪费的、更难维护的，或者违反了哪个 CLAUDE.md 规则），而不是崩溃。当输出上限强制削减时，正确性 bug 始终优先于清理、高度和约定发现。

让每个具有可命名失败场景的候选通过——静默丢弃半信半疑候选的查找器绕过验证步骤，是漏报的主要原因。

## 阶段 2 — 验证（1 票，3 状态）

去重指向同一行/机制的候选，保留具有最具体失败场景的那个。对于每个剩余的候选，通过 Agent 工具运行**一个验证器**：给它 diff、相关文件和候选，并让它准确返回以下之一：

- **CONFIRMED**——可以命名触发它的输入/状态以及错误的输出或崩溃。引用该行。
- **PLAUSIBLE**——机制是真实的，触发是不确定的（时序、环境、配置）。陈述什么可以确认它。
- **REFUTED**——事实错误（代码没有这么说）或在其他地方被守卫。引用证明它的行。

保留投票为 CONFIRMED 或 PLAUSIBLE 的候选。

## 输出

将发现作为最多 8 个对象的 JSON 数组返回：

```json
[
  {
    "file": "path/to/file.ext",
    "line": 123,
    "summary": "one-sentence statement of the bug",
    "failure_scenario": "concrete inputs/state → wrong output/crash"
  }
]
```

按最严重排序。如果超过 8 个存活，保留 8 个最严重的。如果没有通过验证的，返回 `[]`。
