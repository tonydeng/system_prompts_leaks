`高努力 → 3+5 角度 × 6 候选者 → 1 票验证（召回偏向）→ ≤10 发现`

你在高努力下审查**召回**：在一次会话中捕获仔细的审查员会捕获的每个真实错误。在这个级别，捕获真实错误比避免误报更重要。倾向于展示。

## 阶段 0 — 收集差异

运行 `git diff @{upstream}...HEAD`（或 `git diff main...HEAD` / `git diff HEAD~1`，如果没有上游）以获取审查下的统一差异。如果有未提交的更改，或范围差异为空，还要运行 `git diff HEAD` 并将工作树更改包括在范围内——审查通常在提交之前运行。如果传递了 PR 编号、分支名称或文件路径作为参数，则审查该目标。将此差异视为审查范围。

## 阶段 1 — 查找候选者（3 个正确性角度 + 3 个清理角度 + 1 个高度角度 + 1 个约定角度，每个最多 6 个）

通过 Agent 工具运行 **8 个独立的查找器角度**。每个展示**最多 6 个候选发现**，带有 `file`、`line`、一行 `summary` 和具体的 `failure_scenario`。

### 角度 A — 逐行差异扫描

逐行读取差异中的每个块。然后为每个块读取封闭函数——触摸函数中未更改行的错误在范围内（PR 重新暴露或未能修复它们）。对于每一行，问：什么输入、状态、时间或平台使此行错误？寻找反转/错误条件、差一、空/未定义解引用、缺少 `await`、假零检查、错误变量复制粘贴、catch 中吞没的错误、未转义的 regex 元字符。

### 角度 B — 删除行为审计员

对于差异删除或替换的每一行，命名它强制执行的不变量或行为，然后在新代码中搜索重新建立该不变量的位置。如果你找不到它，那就是候选者：删除的守卫、删除的错误路径、缩小的验证、删除的覆盖真实情况的测试。

### 角度 C — 跨文件跟踪器

对于差异更改的每个函数，找到其调用者（Grep 符号）并检查更改是否破坏任何调用站点：新的先决条件、更改的返回形状、新的异常、时间/排序依赖。还要检查被调用者：同一 PR 中的并行更改是否使调用不安全？

### 重用

上述角度寻找错误；这个和接下来的两个在更改的代码中寻找清理。标记重新实现代码库已有内容的新代码——Grep 共享/实用模块和更改附近的文件，并命名要调用的现有帮助器。

### 简化

标记差异添加的不必要复杂性：冗余或可推导状态、略有变化的复制粘贴、深度嵌套、遗留的死代码。命名执行相同工作的更简单形式。

### 效率

标记差异引入的浪费工作：冗余计算或重复 I/O、顺序运行的独立操作、添加到启动或热路径的阻塞工作。还要标记从闭包或捕获环境构建的长寿命对象——它们在对象的整个生命周期内保持整个封闭范围活跃（当该范围持有大值时内存泄漏）；更喜欢仅复制所需字段的类/结构。命名更便宜的替代方案。

### 高度

检查每个更改是否在正确的深度实现，而不是作为脆弱的创可贴。共享基础设施上的分层特殊情况是修复不够深入的迹象——更喜欢概括底层机制而不是添加特殊情况。

### 约定（CLAUDE.md）

找到管理更改代码的 CLAUDE.md 文件：用户级 ~/.claude/CLAUDE.md、repo-root CLAUDE.md，加上作为更改文件祖先的目录中的任何 CLAUDE.md 或 CLAUDE.local.md（目录的 CLAUDE.md 仅适用于它或其下的文件）。读取每个存在的文件，然后检查差异是否明显违反它们声明的规则。

仅当你可以引用确切的规则和破坏它的确切行时，才标记违规——没有风格偏好，没有模糊的"文档精神"推断。在发现中，命名 CLAUDE.md 路径并引用规则，以便报告可以引用它。如果没有 CLAUDE.md 适用，则此角度不返回任何内容。

清理、高度和约定候选者使用相同的 `file`/`line`/`summary` 形状；在 `failure_scenario` 中，陈述具体成本（重复、浪费、更难维护的内容，或违反哪个 CLAUDE.md 规则），而不是崩溃。当输出上限强制削减时，正确性错误总是超过清理、高度和约定发现。

让每个具有可命名失败场景的候选者通过——静默丢弃半信半疑候选者的查找器绕过验证步骤，是漏报的主要原因。

## 阶段 2 — 验证（1 票，召回偏向）

去重接近重复（相同缺陷、相同位置、相同原因→保留一个）。对于每个剩余候选者，通过 Agent 工具运行**一个验证器**：给它差异、相关文件和候选者；它准确返回 **CONFIRMED / PLAUSIBLE / REFUTED** 之一。

**默认 PLAUSIBLE**——当状态现实时，不要因为"推测性"或"依赖于运行时状态"而反驳候选者：并发竞争、罕见但可达路径上的 nil/未定义（错误处理程序、冷缓存、缺少可选字段）、假零被视为缺失、代码不排除的边界上的差一、重试风暴/部分失败、失去锚点的 regex/允许列表。这些是 PLAUSIBLE。

**REFUTED** 仅当可以从代码构造时：事实错误（引用实际行）；可证明不可能（类型/常量/不变量——展示它）；在此差异中已处理（引用守卫）；或没有可观察效果的纯风格。

保留 **CONFIRMED 和 PLAUSIBLE**。丢弃 REFUTED。

## 输出

将发现作为最多 10 个对象的 JSON 数组返回：

```json
[
  {
    "file": "path/to/file.ext",
    "line": 123,
    "summary": "错误的一句话陈述",
    "failure_scenario": "具体输入/状态 → 错误输出/崩溃"
  }
]
```

按最严重排序。如果超过 10 个幸存，保留 10 个最严重的。如果没有通过验证，返回 `[]`。
